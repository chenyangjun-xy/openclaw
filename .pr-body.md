## What Problem This Solves

Fixes the browser plugin lazy-load crash where `openclaw browser status` and any browser agent tool fail with:

```
Package subpath './plugin-sdk/ssrf-runtime-internal' is not defined by "exports"
in openclaw/package.json imported from openclaw\dist\cdp.helpers-B5T7VOSu.js
```

## Why This Change Was Made

`registerManagedProxyBrowserCdpBypass` and `fetchConfiguredLocalOriginWithSsrFGuard` were only exported from `openclaw/plugin-sdk/ssrf-runtime-internal` — a private subpath excluded from `package.json` exports. At runtime Node.js could not resolve the import.

**Fix:** Export both functions from the **public** `openclaw/plugin-sdk/ssrf-runtime` path (already in `package.json` exports), update the two consumer imports, and keep the private module as a backward-compatible re-export shim. The private subpath infrastructure remains intact — only 2 specific exports are promoted.

## User Impact

- `openclaw browser` CLI and browser agent tools work again
- Ollama embedding provider continues to work
- No changes to config, npm package shape, or private subpath behavior

## Root Cause

- Root cause: `registerManagedProxyBrowserCdpBypass` (`fd2a9adbe6`) and `fetchConfiguredLocalOriginWithSsrFGuard` (`a37ebb2d49`) were intentionally placed in the private `ssrf-runtime-internal` module from their creation. The private subpath had no `package.json` exports entry, and its dist artifact was excluded via `!dist/` (added by `2229122077`, "fix: keep private SDK declarations local"). At runtime, Node.js could not resolve imports of this subpath.
- Missing guardrail: the browser plugin lazy-load path was not exercised in CI with a real browser.
